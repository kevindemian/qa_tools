## 1. Color Tokens (Semantic Naming)

### 1.1 Semantic Color Token Architecture

**Claim:** Design systems use a two-tier color token system: primitive (raw palette) tokens and semantic (intent-based) tokens. Semantic tokens abstract purpose from value, enabling theme switching without changing component code.

**Source:** MDN Web Docs - Using CSS custom properties (https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Cascading_variables/Using_custom_properties), last modified Dec 16, 2025.

**Confidence:** HIGH

**Example:**
```css
/* Primitive tokens (raw palette) */
:root {
  --color-blue-500: #3b82f6;
  --color-blue-600: #2563eb;
  --color-green-500: #22c55e;
  --color-red-500: #ef4444;
  --color-gray-50: #f9fafb;
  --color-gray-900: #111827;
}

/* Semantic tokens (intent-based) */
:root {
  --color-surface: var(--color-gray-50);
  --color-surface-elevated: #ffffff;
  --color-text-primary: var(--color-gray-900);
  --color-text-secondary: #6b7280;
  --color-border: #e5e7eb;
  --color-status-success: var(--color-green-500);
  --color-status-error: var(--color-red-500);
  --color-status-info: var(--color-blue-500);
}
```

**Rationale:** Semantic naming decouples *what* a color is used for from *what value* it holds. When themes change, only the semantic layer is remapped. MDN documents this as the canonical use case for CSS custom properties on `:root`.

---

### 1.2 CSS `@property` for Type-Safe Color Tokens

**Claim:** The `@property` at-rule allows registering custom properties with explicit syntax types, initial values, and inheritance control -- enabling validation and predictable fallback for color tokens.

**Source:** MDN Web Docs - Using CSS custom properties, same as above.

**Confidence:** HIGH

**Example:**
```css
@property --color-primary {
  syntax: "<color>";
  inherits: false;
  initial-value: #3b82f6;
}

@property --color-surface {
  syntax: "<color>";
  inherits: true;
  initial-value: #ffffff;
}

/* Usage */
.card {
  background-color: var(--color-surface);
  border-left: 4px solid var(--color-primary);
}
```

**Rationale:** Without `@property`, custom properties accept any value and fail silently at computed time. With `syntax: "<color>"`, invalid assignments are caught at parse time, and `initial-value` provides a guaranteed fallback. This is critical for design token correctness.

## 2. Typography Scale (Modular Scale)

### 2.1 Modular Typography Scale with CSS Custom Properties

**Claim:** A modular type scale uses a consistent ratio (e.g., 1.250 -- Major Third, 1.333 -- Perfect Fourth) to generate font sizes. CSS custom properties encode each step, ensuring mathematical consistency across the entire system.

**Source:** Josh W. Comeau - "The Surprising Truth About Pixels and Accessibility" (https://www.joshwcomeau.com/css/surprising-truth-about-pixels-and-accessibility/), last updated May 15, 2025.

**Confidence:** HIGH

**Example:**
```css
/* Major Third scale (ratio: 1.250) */
:root {
  --font-size-xs: 0.64rem;    /* 10.24px */
  --font-size-sm: 0.8rem;     /* 12.8px */
  --font-size-base: 1rem;     /* 16px (root) */
  --font-size-md: 1.25rem;    /* 20px */
  --font-size-lg: 1.563rem;   /* 25px */
  --font-size-xl: 1.953rem;   /* 31.25px */
  --font-size-2xl: 2.441rem;  /* 39.06px */
  --font-size-3xl: 3.052rem;  /* 48.83px */
}

/* Usage */
h1 { font-size: var(--font-size-2xl); }
h2 { font-size: var(--font-size-xl); }
h3 { font-size: var(--font-size-lg); }
p  { font-size: var(--font-size-base); }
small { font-size: var(--font-size-sm); }
```

**Rationale:** Comeau's article emphasizes that rem units should be used for all typography to respect user font-scaling preferences (accessibility). The CSS variable approach (Comeau's recommended method) lets designers define the scale once and reference it everywhere without error-prone manual calculation.

---

### 2.2 The 62.5% Trick and Its Alternatives

**Claim:** Setting `html { font-size: 62.5%; }` makes 1rem = 10px for mental math convenience, but this breaks third-party components and user expectations. The recommended alternative is to use CSS custom properties or `calc()` expressions.

**Source:** Josh W. Comeau - "The Surprising Truth About Pixels and Accessibility", same as above.

**Confidence:** HIGH

**Example (NOT recommended):**
```css
/* The 62.5% trick -- breaks compatibility */
html { font-size: 62.5%; }
p { font-size: 1.8rem; } /* ~18px */
```

**Example (RECOMMENDED -- CSS variables):**
```css
:root {
  --14px: 0.875rem;
  --15px: 0.9375rem;
  --16px: 1rem;
  --17px: 1.0625rem;
  --18px: 1.125rem;
  --20px: 1.25rem;
  --21px: 1.3125rem;
  --24px: 1.5rem;
}

h1 { font-size: var(--24px); }
p  { font-size: var(--16px); }
```

**Example (RECOMMENDED -- calc):**
```css
h1 { font-size: calc(32rem / 16); } /* = 2rem = 32px */
p  { font-size: calc(16rem / 16); } /* = 1rem = 16px */
```

**Rationale:** The 62.5% trick redefines the global meaning of 1rem, causing tooltip libraries, browser extensions, and user-agent styles to render at 62.5% of expected size. CSS variables named after pixel values or calc expressions preserve the 1rem = 16px baseline.

## 3. Spacing System (4px/8px Grid)

### 3.1 Spacing Scale Design Token Pattern

**Claim:** A spacing scale based on a 4px or 8px grid provides consistent, predictable whitespace. The scale is typically encoded as CSS custom properties using rem values that map to multiples of the base unit.

**Source:** Design systems industry practice (Material Design, Carbon, Primer, etc.), aggregated from Figma open design systems catalog (https://www.designsystems.com/open-design-systems/), accessed July 2026.

**Confidence:** HIGH

**Example:**
```css
/* 4px base spacing scale */
:root {
  --space-0: 0;
  --space-1: 0.25rem;   /* 4px */
  --space-2: 0.5rem;    /* 8px */
  --space-3: 0.75rem;   /* 12px */
  --space-4: 1rem;      /* 16px */
  --space-5: 1.25rem;   /* 20px */
  --space-6: 1.5rem;    /* 24px */
  --space-8: 2rem;      /* 32px */
  --space-10: 2.5rem;   /* 40px */
  --space-12: 3rem;     /* 48px */
  --space-16: 4rem;     /* 64px */
  --space-20: 5rem;     /* 80px */
}

/* Usage */
.card {
  padding: var(--space-4);           /* 16px */
  margin-bottom: var(--space-6);     /* 24px */
  gap: var(--space-2);               /* 8px */
}

.stack > * + * {
  margin-top: var(--space-4);        /* consistent vertical rhythm */
}
```

**Rationale:** Major design systems (Material Design 3, IBM Carbon, GitHub Primer, Atlassian ADS) all use spacing scales based on 4px or 8px increments. The rem-based approach respects user font scaling while maintaining proportional relationships.

---

### 3.2 Spacing with Logical Properties

**Claim:** Modern CSS spacing should use logical properties (`margin-inline`, `padding-block`) instead of physical properties (`margin-left`, `padding-top`) to support bidirectional layouts and maintain consistency.

**Source:** MDN Web Docs - Logical properties and values (https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Logical_properties_and_values), referenced from CSS Guides module list.

**Confidence:** HIGH

**Example:**
```css
.card {
  /* Logical properties (preferred) */
  padding-inline: var(--space-4);
  padding-block: var(--space-3);
  margin-block-end: var(--space-6);

  /* Physical properties (avoid for new code) */
  /* padding: var(--space-3) var(--space-4); */
  /* margin-bottom: var(--space-6); */
}

.stack-inline {
  display: flex;
  gap: var(--space-2);         /* Works regardless of direction */
}
```

**Rationale:** Logical properties map to the document's writing direction, making the spacing system direction-agnostic. For HTML reports (LTR by default), the benefit is primarily future-proofing and consistency with modern CSS best practices.

## 4. Dark Mode with `prefers-color-scheme`

### 4.1 Pure CSS Dark Mode with Custom Properties

**Claim:** Dark mode is best implemented by defining light-mode tokens as default values on `:root`, then redefining the same semantic tokens inside `@media (prefers-color-scheme: dark)`. This requires zero JavaScript for OS-level dark mode detection.

**Source:** MDN Web Docs - prefers-color-scheme (https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/At-rules/@media/prefers-color-scheme), last modified Apr 20, 2026; CSS-Tricks - "Dark Mode in CSS Guide" (https://css-tricks.com/a-complete-guide-to-dark-mode-on-the-web/), Jul 1, 2020.

**Confidence:** HIGH

**Example:**
```css
/* === LIGHT MODE (default) === */
:root {
  --color-surface: #ffffff;
  --color-surface-elevated: #f9fafb;
  --color-surface-card: #ffffff;
  --color-text-primary: #111827;
  --color-text-secondary: #6b7280;
  --color-border: #e5e7eb;
  --color-status-success: #16a34a;
  --color-status-error: #dc2626;
  --color-status-info: #2563eb;
  --color-status-warning: #d97706;
  --shadow-sm: 0 1px 2px rgba(0, 0, 0, 0.05);
  --shadow-md: 0 4px 6px rgba(0, 0, 0, 0.07);
}

/* === DARK MODE === */
@media (prefers-color-scheme: dark) {
  :root {
    --color-surface: #111827;
    --color-surface-elevated: #1f2937;
    --color-surface-card: #1f2937;
    --color-text-primary: #f9fafb;
    --color-text-secondary: #9ca3af;
    --color-border: #374151;
    --color-status-success: #4ade80;
    --color-status-error: #f87171;
    --color-status-info: #60a5fa;
    --color-status-warning: #fbbf24;
    --shadow-sm: 0 1px 2px rgba(0, 0, 0, 0.3);
    --shadow-md: 0 4px 6px rgba(0, 0, 0, 0.4);
  }
}

/* Component code uses ONLY semantic tokens */
.report-card {
  background: var(--color-surface-card);
  color: var(--color-text-primary);
  border: 1px solid var(--color-border);
  box-shadow: var(--shadow-md);
  border-radius: 8px;
  padding: var(--space-4);
}
```

**Rationale:** MDN confirms `prefers-color-scheme` has been baseline widely available since January 2020. CSS-Tricks documents this as the preferred approach for static HTML reports where JavaScript may not be available or desired. The pattern requires zero JS -- the browser handles theme selection based on OS preference.

---

### 4.2 Dark Mode Color Design Principles

**Claim:** Dark mode should NOT use pure black (#000000) backgrounds or pure white (#ffffff) text. Instead, use off-black backgrounds (#121212 to #1f2937) and off-white text (#f9fafb). Desaturated accent colors prevent eye strain.

**Source:** CSS-Tricks - "Dark Mode in CSS Guide" (https://css-tricks.com/a-complete-guide-to-dark-mode-on-the-web/), Jul 1, 2020. Referenced: Material Design Guidelines for dark theme.

**Confidence:** HIGH

**Example:**
```css
/* WRONG: Harsh contrast */
.theme-dark-bad {
  --bg: #000000;
  --text: #ffffff;
  --accent: #ff0000;  /* Fully saturated -- painful on dark */
}

/* CORRECT: Material Design recommended approach */
.theme-dark-good {
  --bg: #121212;           /* Off-black */
  --surface: #1f2937;      /* Elevated surfaces lighter */
  --text: #e5e7eb;         /* Off-white */
  --accent: #ef4444;       /* Desaturated red */
  --accent-subtle: rgba(239, 68, 68, 0.12); /* Accent at 12% opacity for fills */
}
```

**Rationale:** CSS-Tricks (citing Material Design Guidelines) recommends `#121212` as the baseline dark background. The principle is that elevated surfaces should be lighter (not darker) to convey depth via opacity rather than shadow -- inverted from light mode. Desaturated accent colors at lower opacity prevent visual fatigue.

### 4.3 User Override with localStorage Persistence

**Claim:** The complete dark mode pattern combines: (1) `prefers-color-scheme` for OS detection, (2) a body class for manual override, and (3) localStorage for persistence across page loads.

**Source:** CSS-Tricks - "Dark Mode in CSS Guide", same as above.

**Confidence:** HIGH

**Example:**
```css
/* Default: respect OS preference */
:root {
  --color-surface: #ffffff;
  --color-text: #111827;
}

@media (prefers-color-scheme: dark) {
  :root {
    --color-surface: #111827;
    --color-text: #f9fafb;
  }
}

/* User override classes */
body.force-light {
  --color-surface: #ffffff !important;
  --color-text: #111827 !important;
}
body.force-dark {
  --color-surface: #111827 !important;
  --color-text: #f9fafb !important;
}
```

```javascript
// Minimal JS for override + persistence
(function() {
  const saved = localStorage.getItem('theme');
  if (saved === 'light') document.body.classList.add('force-light');
  if (saved === 'dark') document.body.classList.add('force-dark');
})();

function toggleTheme() {
  const body = document.body;
  body.classList.toggle('force-dark');
  body.classList.toggle('force-light');
  localStorage.setItem('theme',
    body.classList.contains('force-dark') ? 'dark' : 'light'
  );
}
```

**Rationale:** CSS-Tricks documents the combined approach as the production-ready pattern. For an HTML report, the OS-level detection alone (zero JS) may suffice, but the override pattern adds user control. The `color-scheme` property on `:root` should also be set to inform UA styles.

## 5. CSS Custom Properties for Theming

### 5.1 Layered Token Architecture

**Claim:** Production CSS design systems use a three-layer token architecture: Global (primitive palette) -> Alias (semantic/intent) -> Component (scoped overrides). This separation enables theming, dark mode, and brand customization without touching component styles.

**Source:** MDN Web Docs - Using CSS custom properties (https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Cascading_variables/Using_custom_properties); Design systems industry practice from Figma open design systems catalog.

**Confidence:** HIGH

**Example:**
```css
/* Layer 1: Global tokens (primitive palette) */
:root {
  --gray-50: #f9fafb;
  --gray-100: #f3f4f6;
  --gray-200: #e5e7eb;
  --gray-300: #d1d5db;
  --gray-400: #9ca3af;
  --gray-500: #6b7280;
  --gray-600: #4b5563;
  --gray-700: #374151;
  --gray-800: #1f2937;
  --gray-900: #111827;

  --blue-500: #3b82f6;
  --green-500: #22c55e;
  --red-500: #ef4444;
  --amber-500: #f59e0b;
}

/* Layer 2: Alias tokens (semantic) */
:root {
  --color-bg-primary: var(--gray-50);
  --color-bg-card: #ffffff;
  --color-bg-hover: var(--gray-100);
  --color-text-body: var(--gray-800);
  --color-text-muted: var(--gray-500);
  --color-border-default: var(--gray-200);
  --color-status-pass: var(--green-500);
  --color-status-fail: var(--red-500);
  --color-status-warn: var(--amber-500);
  --color-status-info: var(--blue-500);
}

/* Layer 3: Component tokens (scoped) */
.report-header {
  --_header-bg: var(--color-bg-card);
  --_header-text: var(--color-text-body);
  --_header-border: var(--color-border-default);
}

.report-header {
  background: var(--_header-bg);
  color: var(--_header-text);
  border-bottom: 1px solid var(--_header-border);
}
```

**Rationale:** The three-layer architecture is practiced by all major design systems (GitHub Primer, Atlassian ADS, Salesforce Lightning, IBM Carbon, etc.). The underscore prefix convention (`--_header-bg`) indicates component-local tokens that are not intended for external use.

### 5.2 `color-scheme` Property for UA Styling

**Claim:** The CSS `color-scheme` property on `:root` tells the browser which color schemes the page supports, causing UA-controlled elements (form controls, scrollbars, selection highlights) to automatically style themselves appropriately.

**Source:** CSS-Tricks - "Dark Mode in CSS Guide" (https://css-tricks.com/a-complete-guide-to-dark-mode-on-the-web/); MDN Web Docs - prefers-color-scheme.

**Confidence:** HIGH

**Example:**
```css
:root {
  color-scheme: light dark;
}
```

**Rationale:** Without this property, form controls and scrollbars may remain in light mode even when the page's custom properties have switched to dark. This is a one-line declaration that prevents a common dark-mode visual inconsistency.

## 6. Component Pattern: Status Indicators for QA Reports

### 6.1 Semantic Status Color System

**Claim:** QA report status indicators (pass/fail/warn/info) should use semantic color tokens with consistent visual treatment: colored left border, tinted background, and colored text/icon.

**Confidence:** MEDIUM

## 7. Accessibility Considerations

### 7.1 Contrast Ratios for Both Themes

**Claim:** All color combinations in both light and dark modes must meet WCAG AA contrast ratios: 4.5:1 for normal text, 3:1 for large text (18px+ bold or 24px+ regular).

**Source:** MDN Web Docs - Accessibility: Color contrast (referenced in CSS Guides > Colors module list).

**Confidence:** HIGH

**Example:**
```css
/* WCAG AA compliant color pairs */

/* Light mode: #111827 on #ffffff = 17.37:1 (AAA) */
/* Light mode: #6b7280 on #ffffff = 5.03:1 (AA)  */

/* Dark mode: #f9fafb on #111827 = 16.75:1 (AAA) */
/* Dark mode: #9ca3af on #111827 = 6.57:1  (AA)  */

/* Status text in dark mode must also meet 4.5:1 */
/* #86efac on #111827 = 10.72:1 (pass text)       */
/* #fca5a5 on #111827 = 8.44:1  (fail text)        */
```

**Rationale:** CSS-Tricks emphasizes that every color pairing must be validated through a contrast checker. The off-white/off-black approach recommended by Material Design naturally produces high contrast ratios. Accent colors used as text must also be validated.
