# F4: Accessibility (WCAG) in Static HTML Reports

**Research Date:** 2026-07-20
**Focus:** WCAG AA HTML reports, accessible HTML dashboard, semantic HTML reports, keyboard navigation HTML, screen reader HTML reports, ARIA attributes reports, accessible data tables

---

## 1. Semantic HTML Landmark Regions for Report Structure

### Claim
WCAG 1.3.1 (Info and Relationships) mandates that page structure be conveyed through semantic HTML elements. Static HTML reports MUST use `<header>`, `<nav>`, `<main>`, `<footer>`, `<article>`, `<section>`, and `<aside>` to define page regions. Screen readers use these landmarks to provide navigation shortcuts (e.g., "jump to main content").

### Source
- **URL:** https://www.w3.org/WAI/tutorials/page-structure/regions/
- **Date:** Updated 24 March 2026
- **Authority:** W3C Web Accessibility Initiative (WAI)

### Confidence
**HIGH** — Primary W3C tutorial with normative WCAG references (SC 1.3.1 Level A, SC 2.4.1 Level A, SC 2.4.6 Level AA).

### Example

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>QA Test Report — Build #4821</title>
</head>
<body>
  <a href="#main-content" class="skip-link">Skip to main content</a>

  <header role="banner">
    <h1>QA Test Report</h1>
    <nav aria-label="Report sections">
      <ul>
        <li><a href="#overview">Overview</a></li>
        <li><a href="#test-results">Test Results</a></li>
        <li><a href="#failures">Failures</a></li>
      </ul>
    </nav>
  </header>

  <main id="main-content">
    <section id="overview" aria-labelledby="overview-heading">
      <h2 id="overview-heading">Overview</h2>
      <!-- report content -->
    </section>

    <section id="test-results" aria-labelledby="results-heading">
      <h2 id="test-results-heading">Test Results</h2>
      <!-- data tables -->
    </section>

    <aside aria-labelledby="meta-heading">
      <h3 id="meta-heading">Metadata</h3>
      <!-- environment info, timestamps -->
    </aside>
  </main>

  <footer role="contentinfo">
    <p>Report generated by QA Tools v2.0</p>
  </footer>
</body>
</html>
```

**Key Pattern:** The HTML5 `<main>` element provides the `main` landmark role. `<nav>` with `aria-label` distinguishes multiple navigation regions. `<aside>` provides complementary information as a separate landmark.

## 2. Skip Links (Bypass Blocks)

### Claim
WCAG 2.4.1 (Bypass Blocks, Level A) requires a mechanism to bypass blocks of content that are repeated on multiple pages. For static HTML reports with navigation headers, a "skip to main content" link is the primary technique. The link MUST be the first focusable element and visually hidden until focused (keyboard users).

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/bypass-blocks.html
- **Date:** Updated 2026 (current WCAG 2.2 Understanding document)
- **Authority:** W3C WCAG 2.2 — Level A requirement

### Confidence
**HIGH** — Normative WCAG 2.2 success criterion. Level A (minimum conformance).

### Example

```css
/* Skip link: hidden visually, visible on focus */
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #1a1a2e;
  color: #ffffff;
  padding: 8px 16px;
  z-index: 100;
  font-size: 1rem;
  text-decoration: none;
  border-radius: 0 0 4px 0;
}

.skip-link:focus {
  top: 0;
}
```

```html
<!-- Must be first focusable element in <body> -->
<body>
  <a href="#main-content" class="skip-link">Skip to main content</a>
  <!-- ...header/nav... -->
  <main id="main-content">
    <!-- report content -->
  </main>
</body>
```

**Key Pattern:** The skip link uses `position: absolute` with negative `top` to be visually off-screen. On `:focus`, it slides into view. This satisfies WCAG 2.4.1 and provides keyboard-only users immediate access to report content.

## 3. Accessible Data Tables with Scope and Headers

### Claim
WCAG 1.3.1 requires data tables to have programmatically determined header-cell/data-cell relationships. The W3C Tables Tutorial specifies three complexity tiers: (a) simple tables use `<th>` with `scope="col"` or `scope="row"`, (b) complex tables with spanning headers use `scope="colgroup"`/`scope="rowgroup"`, and (c) irregular multi-level tables use explicit `id`/`headers` associations. Every data table MUST have a `<caption>` element.

### Source
- **URL:** https://www.w3.org/WAI/tutorials/tables/
- **Date:** Updated 16 February 2023
- **Authority:** W3C WAI Tables Tutorial (normative WCAG references: SC 1.3.1 Level A)

### Confidence
**HIGH** — Primary W3C tutorial. WCAG technique H51 (table markup), H39 (caption), H63 (scope), H43 (id/headers).

### Example — Simple Data Table

```html
<table>
  <caption>Test execution summary for build #4821</caption>
  <thead>
    <tr>
      <th scope="col">Test Suite</th>
      <th scope="col">Passed</th>
      <th scope="col">Failed</th>
      <th scope="col">Skipped</th>
      <th scope="col">Duration</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">Unit Tests</th>
      <td>342</td>
      <td>3</td>
      <td>12</td>
      <td>1m 23s</td>
    </tr>
    <tr>
      <th scope="row">Integration Tests</th>
      <td>128</td>
      <td>1</td>
      <td>5</td>
      <td>4m 56s</td>
    </tr>
    <tr>
      <th scope="row">E2E Tests</th>
      <td>47</td>
      <td>0</td>
      <td>2</td>
      <td>12m 01s</td>
    </tr>
  </tbody>
</table>
```

### Example — Sortable Table with aria-sort

```html
<table aria-label="Test results sorted by failure count">
  <thead>
    <tr>
      <th scope="col" aria-sort="descending">Failures</th>
      <th scope="col">Test Name</th>
      <th scope="col">Category</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>5</td>
      <td>Login authentication flow</td>
      <td>Security</td>
    </tr>
    <!-- ... -->
  </tbody>
</table>
```

**Key Pattern:** `scope` attributes on `<th>` elements create programmatic associations. `<caption>` provides table identification. `aria-sort` indicates current sort state for interactive tables. For non-HTML-table implementations (div-based), use ARIA `role="table"`, `role="row"`, `role="columnheader"`, `role="cell"`.

## 4. ARIA Roles for Non-Semantic Table Structures

### Claim
When data tables are implemented with `<div>` elements (e.g., for CSS Grid layout flexibility), ARIA roles MUST be applied to maintain screen reader accessibility. The WAI-ARIA Table Pattern requires: `role="table"` on the container, `role="row"` on each row, and `role="columnheader"` / `role="rowheader"` / `role="cell"` on cells. For tables with hidden/sorted columns, use `aria-colcount`, `aria-colindex`, `aria-sort`.

### Source
- **URL:** https://www.w3.org/WAI/ARIA/apg/patterns/table/
- **URL:** https://www.w3.org/WAI/ARIA/apg/practices/grid-and-table-properties/
- **Date:** 2026 (current APG documentation)
- **Authority:** W3C ARIA Authoring Practices Guide (APG)

### Confidence
**HIGH** — Primary W3C APG reference with code examples. Normative ARIA specification.

### Example — Div-Based Data Table

```html
<div role="table" aria-label="Test failures by module" aria-rowcount="45">
  <div role="rowgroup">
    <div role="row">
      <span role="columnheader" aria-sort="descending">Module</span>
      <span role="columnheader">Failures</span>
      <span role="columnheader">Last Run</span>
    </div>
  </div>
  <div role="rowgroup">
    <div role="row" aria-rowindex="1">
      <span role="cell">auth-service</span>
      <span role="cell">5</span>
      <span role="cell">2026-07-20 14:32</span>
    </div>
    <div role="row" aria-rowindex="2">
      <span role="cell">payment-gateway</span>
      <span role="cell">3</span>
      <span role="cell">2026-07-20 14:30</span>
    </div>
  </div>
</div>
```

**Key Pattern:** Native HTML `<table>` is strongly preferred over ARIA-div tables (W3C explicitly states this). Use div-based ARIA tables only when CSS Grid layout constraints make `<table>` impractical. `aria-rowcount` / `aria-colcount` communicate total dataset size when virtualizing rows.

## 5. Color Contrast Ratios for Report Text and Data

### Claim
WCAG 1.4.3 (Contrast Minimum, Level AA) requires text and images of text to have a contrast ratio of at least 4.5:1 against their background. Large text (18pt+ or 14pt+ bold) requires 3:1. WCAG 1.4.11 (Non-text Contrast, Level AA) requires 3:1 for UI components and graphical objects (chart borders, status icons). Charts/graphs use 3:1 for axis lines, grid lines, and data series boundaries.

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html
- **Date:** Updated 01 June 2026
- **Authority:** W3C WCAG 2.2 — Level AA requirement

### Confidence
**HIGH** — Normative WCAG 2.2 success criteria. Well-established ratio requirements.

### Example

```css
:root {
  /* WCAG AA compliant color palette */
  --color-text-primary: #1a1a2e;     /* contrast 14.5:1 on white */
  --color-text-secondary: #4a4a6a;   /* contrast 6.8:1 on white */
  --color-bg-primary: #ffffff;
  --color-bg-section: #f8f9fa;
  --color-border: #dee2e6;           /* 2.1:1 — use for decorative only */

  /* Status colors — must be used with text labels, not alone */
  --color-pass: #198754;             /* contrast 4.6:1 on white */
  --color-fail: #dc3545;             /* contrast 5.7:1 on white */
  --color-skip: #6c757d;             /* contrast 4.6:1 on white */
  --color-warning: #fd7e14;          /* contrast 3.1:1 — NOT for text on white */
}

/* Chart data series: use pattern fills + labels, not color alone */
.chart-bar--pass {
  background-color: var(--color-pass);
  /* Add diagonal stripe pattern for additional distinction */
  background-image: repeating-linear-gradient(
    45deg,
    transparent,
    transparent 2px,
    rgba(255,255,255,0.3) 2px,
    rgba(255,255,255,0.3) 4px
  );
}
```

**Key Pattern:** Color-only data encoding is a WCAG 1.4.1 (Use of Color) failure. Charts MUST use pattern fills, shape differences, or text labels alongside color. WCAG 1.4.11 non-text contrast (3:1) applies to chart axis lines, data point borders, and interactive UI elements.

## 6. Keyboard Navigation and Focus Management

### Claim
WCAG 2.1.1 (Keyboard, Level A) requires all functionality operable through a keyboard interface. WCAG 2.4.3 (Focus Order, Level A) requires focusable components receive focus in a meaningful sequence. WCAG 2.4.7 (Focus Visible, Level AA) requires a visible focus indicator. For tabbed report views, focus MUST be managed when switching tabs (roving tabindex or aria-activedescendant pattern).

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/quickref/?currentsidebar=%23col_overview&levels=aa#keyboard-accessible
- **URL:** https://www.w3.org/WAI/ARIA/apg/patterns/tabs/
- **Date:** 2026 (current WCAG 2.2 quick reference)
- **Authority:** W3C WCAG 2.2 + ARIA Authoring Practices Guide

### Confidence
**HIGH** — Normative WCAG 2.2 success criteria. APG provides verified tab patterns.

### Example — Keyboard-Navigable Tabs

```html
<div class="report-tabs">
  <div role="tablist" aria-label="Report views">
    <button role="tab"
            id="tab-overview"
            aria-selected="true"
            aria-controls="panel-overview"
            tabindex="0">
      Overview
    </button>
    <button role="tab"
            id="tab-results"
            aria-selected="false"
            aria-controls="panel-results"
            tabindex="-1">
      Results
    </button>
    <button role="tab"
            id="tab-failures"
            aria-selected="false"
            aria-controls="panel-failures"
            tabindex="-1">
      Failures
    </button>
  </div>

  <div role="tabpanel"
       id="panel-overview"
       aria-labelledby="tab-overview"
       tabindex="0">
    <!-- overview content -->
  </div>

  <div role="tabpanel"
       id="panel-results"
       aria-labelledby="tab-results"
       tabindex="0"
       hidden>
    <!-- results content -->
  </div>

  <div role="tabpanel"
       id="panel-failures"
       aria-labelledby="tab-failures"
       tabindex="0"
       hidden>
    <!-- failures content -->
  </div>
</div>
```

```css
/* WCAG 2.4.7: Visible focus indicator */
[role="tab"]:focus-visible {
  outline: 3px solid #005fcc;
  outline-offset: 2px;
  border-radius: 2px;
}

/* Never remove focus outlines without replacement */
[role="tab"]:focus:not(:focus-visible) {
  outline: none; /* mouse-only, keyboard gets visible ring */
}
```

**Key Pattern:** Roving tabindex pattern: only the active tab has `tabindex="0"`, others have `tabindex="-1"`. Arrow keys move between tabs. Tab key moves focus into the tab panel. `hidden` attribute on inactive panels prevents screen reader access.

## 7. Charts and Graphs — Text Alternatives (Alt Text)

### Claim
WCAG 1.1.1 (Non-text Content, Level A) requires text alternatives for all non-text content. For complex images like charts and graphs, a two-part alternative is required: (a) short `alt` text identifying the chart type and summary, and (b) a long description providing the data in a structured textual format (table, list, or prose). The W3C Images Tutorial provides three approaches for complex image descriptions.

### Source
- **URL:** https://www.w3.org/WAI/tutorials/images/complex/
- **Date:** Updated 08 April 2026
- **Authority:** W3C WAI Images Tutorial (normative WCAG references: SC 1.1.1 Level A)

### Confidence
**HIGH** — Primary W3C tutorial with code examples for charts and graphs.

### Example — Chart with Long Description

```html
<figure role="group">
  <img src="test-trend-chart.png"
       alt="Bar chart showing test pass rates over the last 7 days: Monday 94%, Tuesday 96%, Wednesday 93%, Thursday 97%, Friday 95%, Saturday 98%, Sunday 99%">
  <figcaption>
    <h3>Weekly Test Pass Rate Trend</h3>
    <details>
      <summary>View data table</summary>
      <table>
        <caption>Test pass rates by day</caption>
        <thead>
          <tr>
            <th scope="col">Day</th>
            <th scope="col">Pass Rate</th>
            <th scope="col">Total Tests</th>
          </tr>
        </thead>
        <tbody>
          <tr><th scope="row">Monday</th><td>94%</td><td>482</td></tr>
          <tr><th scope="row">Tuesday</th><td>96%</td><td>491</td></tr>
          <tr><th scope="row">Wednesday</th><td>93%</td><td>478</td></tr>
          <tr><th scope="row">Thursday</th><td>97%</td><td>495</td></tr>
          <tr><th scope="row">Friday</th><td>95%</td><td>487</td></tr>
          <tr><th scope="row">Saturday</th><td>98%</td><td>245</td></tr>
          <tr><th scope="row">Sunday</th><td>99%</td><td>120</td></tr>
        </tbody>
      </table>
    </details>
  </figcaption>
</figure>
```

**Key Pattern:** `<figure>` + `<figcaption>` semantically groups chart with description. `<details>` + `<summary>` provides expandable data table alternative. The `alt` attribute provides a concise summary. The data table provides exact values for screen reader users. This satisfies both SC 1.1.1 and SC 1.3.1.

## 8. Status Icons and Non-Text Differentiation

### Claim
WCAG 1.4.1 (Use of Color, Level A) prohibits using color as the only visual means of conveying information. For test status indicators (pass/fail/warning/skip), each MUST have a text label, icon with alt text, or pattern/shape differentiation alongside color. WCAG 1.4.11 (Non-text Contrast, Level AA) requires 3:1 contrast for these graphical indicators against adjacent colors.

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html
- **Date:** 2026 (current WCAG 2.2)
- **Authority:** W3C WCAG 2.2 — Level A and Level AA requirements

### Confidence
**HIGH** — Normative WCAG 2.2 success criteria.

### Example

```html
<!-- CORRECT: Multiple differentiators -->
<span class="status-badge status-pass">
  <svg aria-hidden="true" class="icon">
    <use href="#icon-check"></use>
  </svg>
  <span class="sr-only">Passed</span>
  Passed
</span>

<span class="status-badge status-fail">
  <svg aria-hidden="true" class="icon">
    <use href="#icon-x"></use>
  </svg>
  <span class="sr-only">Failed</span>
  Failed
</span>

<!-- WRONG: Color only -->
<span style="background: green">342</span>  <!-- FAILURE: WCAG 1.4.1 -->
```

```css
/* Screen reader only class */
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

.status-badge {
  display: inline-flex;
  align-items: center;
  gap: 4px;
  padding: 2px 8px;
  border-radius: 4px;
  font-weight: 600;
}

.status-pass {
  background-color: #d4edda;
  color: #155724;
  border-left: 4px solid #198754;
}

.status-fail {
  background-color: #f8d7da;
  color: #721c24;
  border-left: 4px solid #dc3545;
}
```

**Key Pattern:** Each status uses color + text label + border-left shape indicator. The `border-left` provides a shape differentiator independent of color. SVG icons carry `aria-hidden="true"` (decorative) with adjacent text labels providing the accessible name.

## 9. Language Declaration and Heading Hierarchy

### Claim
WCAG 3.1.1 (Language of Page, Level A) requires the `lang` attribute on `<html>`. WCAG 2.4.6 (Headings and Labels, Level AA) requires headings to describe topic or purpose. WCAG 2.4.10 (Section Headings, Level AAA, advisory for AA) recommends headings at the beginning of each content section. Heading hierarchy MUST NOT skip levels (h1 -> h3 violates structure). Technique H69 (Sufficient for 2.4.1 and 2.4.10) provides section headings for bypass blocks.

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Techniques/html/H69
- **Date:** Updated 12 January 2026
- **Authority:** W3C WCAG 2.2 Technique — Sufficient for SC 2.4.1 and SC 2.4.10

### Confidence
**HIGH** — Normative WCAG technique with code examples.

### Example

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>QA Report — Build #4821 — 2026-07-20</title>
</head>
<body>
  <!-- h1: unique page identifier -->
  <h1>QA Test Report — Build #4821</h1>

  <main>
    <!-- h2: major report sections -->
    <section aria-labelledby="sec-overview">
      <h2 id="sec-overview">Overview</h2>
      <p>Build executed on 2026-07-20 at 14:32 UTC</p>

      <!-- h3: subsections within Overview -->
      <h3>Environment</h3>
      <p>Node.js v20.11.0, TypeScript 5.4</p>
    </section>

    <section aria-labelledby="sec-results">
      <h2 id="sec-results">Test Results</h2>

      <h3>Unit Tests</h3>
      <p>342 passed, 3 failed, 12 skipped</p>

      <h3>Integration Tests</h3>
      <p>128 passed, 1 failed, 5 skipped</p>
    </section>

    <section aria-labelledby="sec-failures">
      <h2 id="sec-failures">Failure Details</h2>

      <h3>auth-service</h3>
      <p>5 failures detected</p>

      <h3>payment-gateway</h3>
      <p>1 failure detected</p>
    </section>
  </main>
</body>
</html>
```

**Key Pattern:** `lang="en"` enables correct screen reader pronunciation. Heading hierarchy (h1 > h2 > h3) creates a navigable document outline. Screen readers allow users to jump between headings. Each section starts with its heading (H69 technique).

## 10. Live Regions for Dynamic Report Updates

### Claim
For reports that update in real-time (e.g., live CI/CD dashboards), WCAG 4.1.3 (Status Messages, Level AA) requires that status messages be programmatically determinable through role or properties without receiving focus. Use `role="status"` for passive updates and `role="alert"` for critical failures. `aria-live="polite"` is the general-purpose mechanism for announcing content changes.

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html
- **Date:** 2026 (current WCAG 2.2)
- **Authority:** W3C WCAG 2.2 — Level AA requirement

### Confidence
**HIGH** — Normative WCAG 2.2 success criterion.

### Example

```html
<!-- Live status banner -->
<div role="status" aria-live="polite">
  <p>Test execution in progress... 234/482 tests completed (48.5%)</p>
</div>

<!-- Critical failure notification -->
<div role="alert">
  <p>3 critical failures detected in auth-service module</p>
</div>

<!-- Progress bar with accessible text -->
<div role="progressbar"
     aria-valuenow="234"
     aria-valuemin="0"
     aria-valuemax="482"
     aria-label="Test execution progress">
  <div class="progress-fill" style="width: 48.5%"></div>
  <span class="sr-only">234 of 482 tests completed</span>
</div>
```

**Key Pattern:** `role="status"` has implicit `aria-live="polite"` — screen readers announce changes after the current speech finishes. `role="alert"` has implicit `aria-live="assertive"` — announced immediately. `role="progressbar"` provides structured progress information. For static reports (no live updates), this pattern is not required but demonstrates best practice for report generators that may produce live dashboards.

## 11. Text Spacing and Readability (Cognitive Accessibility)

### Claim
WCAG 1.4.12 (Text Spacing, Level AA) requires that text can be resized and reflowed without loss of content or functionality. Users MUST be able to override: text spacing to 1.5em, paragraph spacing to 2em, letter spacing to 0.12em, word spacing to 0.16em. WCAG Technique C21 recommends `line-height: 1.5` as a baseline for body text to support cognitive accessibility.

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/text-spacing.html
- **URL:** https://www.w3.org/WAI/WCAG22/Techniques/css/C21
- **Date:** Updated 12 January 2026 (C21), 2026 (text-spacing understanding)
- **Authority:** W3C WCAG 2.2 — Level AA requirement + Sufficient Technique

### Confidence
**HIGH** — Normative WCAG 2.2 success criterion + sufficient technique.

### Example

```css
/* Base report typography */
body {
  font-family: system-ui, -apple-system, 'Segoe UI', Roboto, sans-serif;
  font-size: 1rem;           /* 16px base */
  line-height: 1.5;          /* C21: sufficient for 1.4.12 */
  color: #1a1a2e;
  background-color: #ffffff;
}

/* Ensure text spacing overrides work (WCAG 1.4.12) */
/* Do NOT use fixed heights or overflow: hidden on text containers */
.report-content p {
  margin-bottom: 1em;        /* Allows paragraph spacing override */
}

/* Allow reflow at 320px viewport width (WCAG 1.4.10 Reflow) */
@media (max-width: 320px) {
  .report-content {
    font-size: 1rem;
    /* Content reflows, no horizontal scrolling needed */
  }
}
```

**Key Pattern:** Avoid `overflow: hidden`, fixed `height`, or `max-height` on text containers — these prevent users from applying WCAG 1.4.12 text spacing overrides. Use relative units (`rem`, `em`) for font sizes. `line-height: 1.5` is the minimum recommended for body text (C21 technique).

## 12. Accessible SVG Charts (Inline SVG)

### Claim
When charts are rendered as inline SVG (e.g., D3.js, Chart.js), the SVG element MUST have a `<title>` element (equivalent to `alt` for images) and optionally a `<desc>` element for long description. The SVG MUST have `role="img"` or `role="graphics-document"` and `aria-labelledby` pointing to the title. For decorative SVG, use `aria-hidden="true"`.

### Source
- **URL:** https://www.w3.org/WAI/tutorials/images/complex/
- **URL:** https://www.w3.org/TR/svg-aam-1.0/
- **Date:** 2026 (current W3C resources)
- **Authority:** W3C WAI + SVG Accessibility API Mappings

### Confidence
**MEDIUM** — WAI Images Tutorial covers SVG alt text. SVG-AAM is a W3C Working Draft (not yet Recommendation). Browser/screen reader support for `role="graphics-document"` is evolving.

### Example

```html
<figure role="group">
  <svg role="img"
       aria-labelledby="chart-title chart-desc"
       width="600" height="400"
       viewBox="0 0 600 400">
    <title id="chart-title">Test pass rate by module</title>
    <desc id="chart-desc">Bar chart showing test pass rates: auth-service 94%, api-gateway 98%, payment-gateway 97%, user-service 99%</desc>

    <!-- Chart rendering -->
    <rect x="50" y="50" width="100" height="300" fill="#198754" aria-label="auth-service: 94%" />
    <rect x="180" y="20" width="100" height="330" fill="#198754" aria-label="api-gateway: 98%" />
    <!-- ... -->
  </svg>
  <figcaption>
    <p>Module pass rates for build #4821</p>
  </figcaption>
</figure>
```

**Key Pattern:** `<title>` provides the short description (announced by screen readers). `<desc>` provides the long description. `aria-labelledby` references both. Individual data elements can carry `aria-label` for drill-down accessibility. `role="img"` prevents screen readers from traversing internal SVG structure.

## 13. Focus Appearance and Target Size (WCAG 2.2 New Criteria)

### Claim
WCAG 2.4.13 (Focus Appearance, Level AAA) requires focus indicators to have an area at least as large as the perimeter of the element times 2 CSS pixels, with a 3:1 contrast ratio between focused and unfocused states. WCAG 2.5.8 (Target Size Minimum, Level AA) requires interactive targets to be at least 24x24 CSS pixels. These apply to report navigation links, tab buttons, and expandable sections.

### Source
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/focus-appearance.html
- **URL:** https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html
- **Date:** 2026 (WCAG 2.2)
- **Authority:** W3C WCAG 2.2 — Level AA (2.5.8) and Level AAA (2.4.13)

### Confidence
**MEDIUM** — WCAG 2.2 is a W3C Recommendation. Focus Appearance (2.4.13) is Level AAA (advisory, not required for AA conformance). Target Size (2.5.8) is Level AA (required).

### Example

```css
/* WCAG 2.5.8: Minimum 24x24px target size */
.nav-link {
  display: inline-flex;
  align-items: center;
  padding: 8px 16px;          /* Effective target > 24x24px */
  min-height: 44px;           /* WCAG 2.5.5 Enhanced: 44x44px */
  min-width: 44px;
  font-size: 1rem;
  text-decoration: none;
  color: #1a1a2e;
  border: 2px solid transparent;
}

/* WCAG 2.4.13: Focus appearance */
.nav-link:focus-visible {
  outline: 3px solid #005fcc;
  outline-offset: 2px;
  background-color: #e8f0fe;
}

/* Ensure 3:1 contrast between focused and unfocused states */
/* Unfocused: #ffffff background, Focused: #e8f0fe background */
/* Difference in relative luminance meets 3:1 requirement */
```

**Key Pattern:** WCAG 2.5.8 (Level AA) is mandatory for interactive report elements. Minimum 24x24px touch/click target. WCAG 2.4.13 (Level AAA) is aspirational — use `outline` + `outline-offset` for visible focus rings that exceed the 2px CSS pixel minimum perimeter requirement.
